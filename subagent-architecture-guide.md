# 3-19 | Архитектура субагентов — Руководство

> Как проектировать команду AI-субагентов для Claude Code.
> Роли, модели, коммуникация, типовые составы команд.

---

## Философия этого документа

Этот файл — **справочник для проектирования команд субагентов**. Вы не пишете конфиги субагентов вручную — вы используете его как:

1. **Шаблон архитектурных решений** — какая команда нужна под ваш тип проекта (SaaS, бот, AI-агент)
2. **Reference для Claude** — загрузить в проект, чтобы Claude Code при bootstrap сам создал правильных субагентов
3. **Валидация результата** — проверить, что Claude выбрал правильные модели (Opus для БД, Sonnet для UI) и корректные ограничения инструментов

**Принцип работы:**
- Вы — архитектор: решаете, нужна ли расширенная команда (5 агентов) или базовая (4)
- Claude Code — исполнитель: создаёт файлы `.claude/agents/*.md` по правилам из этого документа
- Этот документ — общий язык между вами

**Как применять:**
```
Сценарий 1: Claude создаёт команду под проект
  Вы → даёте Claude описание проекта + этот файл → Claude генерирует всех субагентов

Сценарий 2: Вы валидируете созданную команду
  Вы → открываете `.claude/agents/` → сверяете с типовыми составами из этого файла

Сценарий 3: Нестандартный проект
  Вы → выбираете подходящий состав → просите Claude собрать команду с кастомизациями
```

## Зачем нужны субагенты

Claude Code по умолчанию — один универсальный агент. Он умеет всё, но при сложных проектах теряет фокус: путает стили кода для фронта и бэка, забывает про RLS, пишет миграции внутри компонентов.

**Субагенты** решают это через специализацию:
- Каждый агент отвечает за свою зону
- У каждого свои правила и ограничения
- Главный агент (оркестратор) делегирует задачи нужному субагенту

```
Вы (промпт) ──► Оркестратор (Claude Code)
                    ├── database-architect   (Opus)
                    ├── backend-engineer     (Sonnet)
                    ├── frontend-developer   (Sonnet)
                    └── qa-reviewer          (Sonnet)
```

## Стандартная команда: 4 субагента

Подходит для 80% SaaS-проектов. Используйте по умолчанию.

### 1. database-architect

| Параметр | Значение |
|----------|----------|
| **Модель** | Opus |
| **Зона** | Схема БД, миграции, RLS-политики, индексы, seed-скрипты |
| **Инструменты** | Read, Write, Edit, Bash, Glob, Grep |
| **Когда вызывать** | Создание таблиц, изменение схемы, настройка RLS, оптимизация запросов |

**Почему Opus:** Архитектура базы данных — фундамент проекта. Ошибка здесь каскадно ломает всё остальное. Opus лучше справляется со сложными связями между таблицами, RLS-политиками и оптимизацией индексов.

### 2. backend-engineer

| Параметр | Значение |
|----------|----------|
| **Модель** | Sonnet |
| **Зона** | API routes, бизнес-логика, интеграции, серверные действия |
| **Инструменты** | Read, Write, Edit, Bash, Glob, Grep |
| **Когда вызывать** | Создание API, server actions, подключение внешних сервисов, обработка данных |

**Почему Sonnet:** API-роуты и бизнес-логика — объёмная, но структурно простая работа. Sonnet справляется отлично и стоит значительно дешевле Opus.

### 3. frontend-developer

| Параметр | Значение |
|----------|----------|
| **Модель** | Sonnet |
| **Зона** | React-компоненты, страницы, стилизация, responsive |
| **Инструменты** | Read, Write, Edit, Bash, Glob, Grep |
| **Когда вызывать** | Создание UI, стилизация, интерактивные элементы, адаптивная вёрстка |

**Почему Sonnet:** UI-компоненты — повторяющиеся паттерны. Sonnet пишет чистый React + Tailwind быстро и дёшево.

### 4. qa-reviewer

| Параметр | Значение |
|----------|----------|
| **Модель** | Sonnet |
| **Зона** | Тестирование, код-ревью, безопасность, проверка типов |
| **Инструменты** | Read, Bash, Glob, Grep (без Write — только читает и тестирует) |
| **Когда вызывать** | После реализации модуля, перед деплоем, при подозрении на баг |

**Почему Sonnet:** Ревью — это поиск паттернов ошибок. Sonnet хорошо находит типичные проблемы. Обратите внимание: у QA **нет Write** — он не должен менять код, только указывать на проблемы.

## Расширенная команда: 5 субагентов

Добавляется **пятый агент** когда в проекте есть production AI-компоненты (LLM-вызовы, RAG, AI-агенты).

### 5. ai-agent-architect (дополнительный)

| Параметр | Значение |
|----------|----------|
| **Модель** | Opus |
| **Зона** | Промпты, RAG-пайплайны, LLM-интеграции, оценка качества AI |
| **Инструменты** | Read, Write, Edit, Bash, Glob, Grep |
| **Когда вызывать** | Проектирование AI-фич, настройка промптов, создание RAG, оценка точности |

**Почему Opus:** AI-архитектура требует глубокого понимания возможностей и ограничений LLM. Ошибки в промптах и RAG-пайплайнах сложно отлавливать — здесь экономия на модели дорого обойдётся.

## Расширенная библиотека субагентов

Базовые 4-5 агентов покрывают типовой SaaS. Но под реальные задачи часто нужны специализированные агенты. Вот каталог, проверенный на практике.

### Разработка и DevOps

**devops-engineer** (Sonnet)
- **Зона:** VPS-настройка, Docker, PM2, Nginx, SSL, CI/CD, мониторинг
- **Когда:** Деплой, настройка production-окружения, миграция между серверами
- **Инструменты:** Bash, Read, Write, Edit
- **Пример description:** "Настраивает production-инфраструктуру на Beget VPS. ИСПОЛЬЗУЙ для деплоя, Docker, Nginx, SSL."

**security-auditor** (Opus)
- **Зона:** OWASP-аудит, RLS-политики, env-vars, SQL injection, XSS, CSRF
- **Когда:** Перед production-деплоем, после крупных изменений auth/API
- **Инструменты:** Read, Bash, Glob, Grep (без Write)
- **Почему Opus:** цена ошибки в безопасности максимальна

**performance-optimizer** (Sonnet)
- **Зона:** Индексы БД, кеширование, bundle size, Lighthouse, N+1 queries
- **Когда:** Перед запуском, при жалобах на скорость, при росте нагрузки
- **Инструменты:** Read, Edit, Bash, Glob, Grep

**migration-specialist** (Opus)
- **Зона:** Миграции между стеками (Next.js 15→16, Tailwind v3→v4), breaking changes
- **Когда:** Обновление мажорных версий, рефакторинг legacy-кода
- **Инструменты:** Read, Write, Edit, Bash

### AI и данные

**prompt-engineer** (Opus)
- **Зона:** Системные промпты, few-shot примеры, chain-of-thought, тестирование
- **Когда:** Проектирование AI-агентов, улучшение качества ответов LLM
- **Отличие от ai-agent-architect:** prompt-engineer — чистое написание промптов, ai-agent-architect — архитектура целой AI-системы (RAG, tools, agents)

**data-analyst** (Sonnet)
- **Зона:** SQL-аналитика, дашборды, отчёты, визуализация (Recharts)
- **Когда:** Построение аналитических разделов, метрики, BI
- **Инструменты:** Read, Write, Edit, Bash, Glob, Grep

**rag-engineer** (Opus)
- **Зона:** pgvector, embeddings, chunking, semantic search, context injection
- **Когда:** Knowledge base для AI-агента, поиск по документам, RAG-пайплайны

### Контент и методология (из нашей практики)

**content-writer** (Sonnet)
- **Зона:** Генерация текстового контента из источников (уроки, статьи, методички)
- **Когда:** Массовая генерация контента по шаблонам и методологии
- **Использовался в:** генерации 57 toolkit-документов этого курса

**methodology-reviewer** (Opus)
- **Зона:** Валидация контента против source-of-truth документа, поиск устаревших подходов
- **Когда:** Проверка учебных материалов, документации, методичек на актуальность
- **Использовался в:** проверке METHODOLOGY_2026.md vs фактических уроков

**technical-validator** (Sonnet)
- **Зона:** Проверка кода через Context7 MCP, валидация API-синтаксиса, версий библиотек
- **Когда:** Перед публикацией технического контента или кода
- **Связка с MCP:** всегда использует Context7 для проверки актуальности

**nda-reviewer** (Sonnet)
- **Зона:** Поиск утечек клиентских данных: имена компаний, URL, API-ключи, суммы
- **Когда:** Перед публикацией документов, извлечённых из реальных проектов
- **Использовался в:** очистке toolkit-документов курса от NDA-данных

### Маркетинг и продажи

**copywriter** (Sonnet)
- **Зона:** Лендинги (VORTEX, Hormozi), email-воронки, посты, хуки для Reels
- **Когда:** Маркетинговые тексты, КП, описания продуктов

**seo-specialist** (Sonnet)
- **Зона:** Meta-tags, structured data, Lighthouse SEO, семантическое ядро
- **Когда:** Подготовка к индексации, аудит существующих страниц

**sales-analyst** (Opus)
- **Зона:** SPIN-квалификация лидов, анализ звонков, скрипты переговоров, unit-экономика
- **Когда:** Работа с клиентским pipeline, воронка продаж

### Юридика и compliance

**legal-reviewer** (Opus)
- **Зона:** 152-ФЗ, GDPR, политики конфиденциальности, оферты, договоры
- **Когда:** Перед запуском в production, при работе с ПД пользователей
- **Почему Opus:** юридические формулировки требуют высокой точности

## Скиллы как расширение субагентов

В Claude Code есть **скиллы** — это не субагенты, но они дополняют их. Скилл = специализированная команда (`/название`), которая запускает готовую процедуру.

### Разница между субагентом и скиллом

| | Субагент | Скилл |
|---|----------|-------|
| **Вызов** | Автоматически оркестратором (по description) | Явно пользователем (`/название`) |
| **Контекст** | Своя модель, своя зона, свои инструменты | Работает в текущем контексте |
| **Применение** | Делегирование большой задачи | Выполнение процедуры по чек-листу |
| **Пример** | "backend-engineer напиши API" | `/deploy` — запустить pipeline деплоя |

### Скиллы из этого курса

Мы создали глобальные скиллы, которые работают во всех проектах:

- **`/deploy`** — деплой с type-check, верификацией target, smoke-тестами
- **`/debug-api`** — систематическая отладка API-интеграций
- **`/connect-supabase`** — надёжное подключение Supabase MCP
- **`/handoff`** — генерация HANDOFF.md для передачи сессии
- **`/new-project`** — бутстрап нового проекта с CLAUDE.md
- **`/course-lesson`** — генерация урока из методологии (для курса)
- **`/review-methodology`** — проверка контента на актуальность
- **`/generate-template`** — создание чистых шаблонов из реальных документов

### Комбинирование: субагент + скилл

```
Пример: backend-engineer вызывается для новой фичи
        ↓
        пишет API endpoint
        ↓
        пользователь запускает /debug-api (скилл)
        ↓
        скилл проводит по чек-листу проверки
        ↓
        qa-reviewer (субагент) валидирует итог
```

Субагент = "кто" (специалист). Скилл = "как" (процедура). Используйте оба.

## Как просить Claude создавать субагентов под проект

**Главная мысль этого раздела:** вы не обязаны создавать файлы субагентов вручную. Claude Code сам может проектировать и генерировать их под любой проект.

### Промпт для создания субагента с нуля

```
Создай субагента для задачи: [описание задачи].

Контекст проекта: [краткое описание проекта]
Стек: [перечислить]
Модель: [opus/sonnet — по сложности]

Требования к субагенту:
- Чёткая зона ответственности (не пересекается с [перечислить существующих])
- Description с "ИСПОЛЬЗУЙ для..."
- Список инструментов под задачу
- Принципы работы (3-5 пунктов)
- Context7-ссылки на релевантные библиотеки

Сохрани в .claude/agents/[имя].md
```

### Промпт для проектирования всей команды

```
Спроектируй команду субагентов для этого проекта.

Прочитай SPEC.md и определи:
1. Сколько агентов нужно (минимум 3, максимум 7)
2. Какие модели назначить (Opus для архитектуры, Sonnet для реализации)
3. Как разделить зоны ответственности без пересечений
4. Какие Context7-библиотеки релевантны каждому

Создай все файлы в .claude/agents/
Затем создай .claude/CLAUDE.md с описанием команды.
```

### Промпт для аудита существующей команды

```
Проанализируй текущих субагентов в .claude/agents/.

Проверь:
1. Нет ли пересечений зон ответственности
2. Правильно ли назначены модели (не Opus везде)
3. У QA нет Write?
4. Все ли description содержат "ИСПОЛЬЗУЙ для..."
5. Нужны ли дополнительные агенты под задачи из SPEC.md

Предложи улучшения и создай недостающих агентов.
```

### Мета-паттерн: Claude создаёт Claude-агентов

Это ключевое понимание. Claude Code умеет:
- **Читать** существующих агентов
- **Анализировать** их эффективность
- **Проектировать** новых под новые задачи
- **Создавать** `.md`-файлы агентов
- **Обновлять** их при изменении требований

Вы — архитектор команды. Claude Code — тот, кто физически создаёт членов команды по вашим спецификациям.

```
Вы:  "В проекте добавляется e-commerce. Нужен ecommerce-specialist."
      ↓
Claude Code:  Создаёт .claude/agents/ecommerce-specialist.md
              (Shopify/WooCommerce API, корзина, checkout, ЮKassa)
      ↓
Вы:  "Добавь в команду."
      ↓
Claude Code:  Обновляет CLAUDE.md с новым членом команды
```

## Выбор модели: Opus vs Sonnet

### Когда использовать Opus

- Архитектурные решения (схема БД, системный дизайн)
- Сложные алгоритмы с множеством edge-cases
- AI/LLM интеграции (промпт-инжиниринг, RAG)
- Задачи с высокой ценой ошибки (безопасность, финансы)

### Когда использовать Sonnet

- Написание кода по готовой архитектуре
- CRUD-операции, API-роуты
- UI-компоненты по готовым макетам
- Тестирование и код-ревью
- Повторяющиеся паттерны

### Стоимостные ориентиры

| Модель | Стоимость (вход) | Стоимость (выход) | Скорость |
|--------|:-----------------:|:-----------------:|:--------:|
| **Opus** | ~$15/M tokens | ~$75/M tokens | Медленнее |
| **Sonnet** | ~$3/M tokens | ~$15/M tokens | Быстрее |

**Практический расчёт:** в типичном SaaS-проекте ~80% работы делают Sonnet-агенты (бэкенд, фронт, QA), и ~20% — Opus (БД-архитектура). Это даёт экономию ~60% по сравнению с "Opus на всё".

## Коммуникация между субагентами

### Паттерн: последовательная делегация

Оркестратор вызывает субагентов в определённом порядке. Каждый следующий агент видит результат предыдущего.

```
Оркестратор
    │
    ├── 1. database-architect ──► Создаёт таблицы, миграции
    │       ↓ (файлы на диске)
    ├── 2. backend-engineer ──► Пишет API, читая схему БД
    │       ↓ (файлы на диске)
    ├── 3. frontend-developer ──► Пишет UI, вызывая API
    │       ↓ (файлы на диске)
    └── 4. qa-reviewer ──► Тестирует всё вместе
            ↓ (отчёт оркестратору)
```

### Принципы коммуникации

1. **Агенты не общаются напрямую** — только через оркестратора
2. **Общий контекст — файловая система** — агент читает файлы, созданные другими
3. **Чёткие зоны ответственности** — один файл редактирует только один агент
4. **QA никогда не правит код** — только сообщает о проблемах

## Структура файлов субагента

Каждый субагент — это `.md` файл в `.claude/agents/`:

```
.claude/
└── agents/
    ├── database-architect.md
    ├── backend-engineer.md
    ├── frontend-developer.md
    └── qa-reviewer.md
```

### Формат файла субагента

```markdown
---
name: имя-агента
description: "Краткое описание. ИСПОЛЬЗУЙ для [тип задач]."
tools: Read, Write, Edit, Bash, Glob, Grep
model: opus | sonnet
---

Ты — [роль]. [Контекст проекта].

## Зона ответственности
- Пункт 1
- Пункт 2

## Принципы
- Правило 1
- Правило 2

## Ключевые файлы
- path/to/file — описание

## Context7
- use context7 для [библиотеки]
```

**Важно в description:** фраза "ИСПОЛЬЗУЙ для..." помогает оркестратору правильно выбирать агента для задачи.

## Пример: команда для проекта TaskFlow

**Проект:** TaskFlow — трекер задач для фрилансеров
**Компания:** ООО "Продуктив"
**Стек:** Next.js 16, Supabase, Tailwind v4, ЮKassa
**Тип:** SaaS (типовой) → стандартная команда из 4 агентов

## Частые ошибки при настройке субагентов

| Ошибка | Почему плохо | Как правильно |
|--------|-------------|---------------|
| Opus на все агенты | Дорого, медленно, не нужно | Opus только для БД и AI-архитектуры |
| Один универсальный агент | Теряет контекст, путает зоны | Минимум 3-4 специализированных |
| QA с правом Write | Начинает править код сам | QA только Read + Bash (тесты) |
| Нет description с "ИСПОЛЬЗУЙ для..." | Оркестратор не знает кого вызвать | Всегда явно указывать триггер |
| Слишком детальные инструкции | Агент тратит контекст на чтение | 30-60 строк на агента, не больше |
| Дублирование зон | Два агента правят один файл | Чёткое разграничение ответственности |

## Практическое упражнение

**Задача:** спроектировать команду для вашего реального проекта.

### Шаг 1: определите тип проекта
Сверьтесь с матрицей выше. Нашли близкий тип? Используйте его как старт.

### Шаг 2: попросите Claude создать команду
Скопируйте промпт "Спроектируй команду субагентов для этого проекта" (см. выше) и замените переменные на ваш контекст.

### Шаг 3: проведите аудит
После создания запустите промпт аудита. Claude проверит свою же работу.

### Шаг 4: добавляйте по мере роста проекта
Не нужно сразу проектировать 10 агентов. Начните с 4. Когда столкнётесь с новой зоной (деплой, безопасность, контент) — попросите Claude создать соответствующего специалиста.

**Главное правило:** субагенты — это живая команда. Она растёт вместе с проектом. Вы не обязаны знать всех специалистов заранее — вы знаете, как попросить Claude их создать.

## Бесконечная композиция: мысленная модель

Представьте компанию:
- **CEO** — это вы (архитектор, коммуникатор)
- **Руководитель отдела** — Claude Code (оркестратор)
- **Отделы** — субагенты (специалисты по зонам)
- **Процедуры** — скиллы (стандартные операции)
- **Knowledge base** — vault + Context7 (источник актуальных знаний)

Вы не нанимаете каждого сотрудника лично. Вы говорите руководителю: "нам нужен специалист по X". Он находит/создаёт нужного человека, обучает, интегрирует в процессы.

В Claude Code это работает так же:
- Вы описываете задачу на человеческом языке
- Claude Code определяет, нужен ли новый субагент
- Если нужен — создаёт его, добавляет в команду, делегирует задачу
- Результат возвращается через оркестратора

**Вам не нужно быть программистом.** Вам нужно уметь описывать задачи и спрашивать правильные вопросы.
